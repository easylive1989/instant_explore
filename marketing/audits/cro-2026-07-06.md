# CRO 稽核報告：Lorescape 下載 + 訂閱漏斗

稽核日期：2026-07-06
GA4 資料範圍：2026-06-30 – 2026-07-05（GA4_PROPERTY_ID_WEB + APP，via lorescape-metrics）
其他來源：App Store Connect（瀏覽器）、Play Console（瀏覽器）、RevenueCat 快照、IG/IG_posts、GSC、落地頁原始碼（`landing/src/i18n/dictionaries.ts`）

---

## 一句話結論

**這是一個「前流量」App，不是「低轉換率」App。** 每天約 1 位訪客進落地頁、
IG profile_views 近兩日為 0、安裝與訂閱近零。此時投資「落地頁轉換率微調」屬低槓桿；
綁定限制在 **① 曝光→網站/商店的流量本身、② IG reach→profile→連結的漏水**。
落地頁與商店頁仍有結構性洞（無定價/試用透明度、無異議處理、無信任證明、無歸因追蹤），
應趁流量還沒起來時補好。

落地頁 → 安裝的逐段轉換 **無法量測**（商店連結未帶追蹤參數）——這本身是缺陷。

## 5 層分析

### Layer 1 技術效能（先修）
- ~~商店連結缺歸因~~ **（已修 2026-07-06）**：站內點擊事件 `download_click`
  (platform+location) 本就存在；本次補上**商店端安裝歸因**——Play install
  referrer（UTM）＋App Store campaign token `ct=web_<location>`，見
  `landing/src/lib/downloadLinks.ts`。現在五個 CTA（navbar/hero/final_cta/footer/story）
  皆可在 Play Console / App Analytics 依區塊歸因安裝。
- 落地頁 Core Web Vitals 未實測（本次未跑 PageSpeed）→ `missing_data`，建議補測行動裝置 LCP。
- App 首啟與 RevenueCat paywall 渲染崩潰率：`missing_data`（安裝樣本太少無法判讀）。

### Layer 2 流量與受眾品質
- 到落地頁的流量近乎為零；主要曝光在 IG，但 **reach→profile_views≈1% 且近兩日 0**。
  問題不在落地頁能不能轉換，而在「觀眾根本沒被導去任何落點」。
- IG Reel 內容（景點故事）與 App value prop 高度一致（scent trail 良好），
  但**缺少把觀眾推向 bio 連結 / 商店的明確 CTA**——與 reels-planner 8/3 檢核點所記一致。

### Layer 3 Offer 與定價（高槓桿、目前最弱）
- 落地頁**從頭到尾沒有價格、沒有方案、沒有試用、沒有退款/風險反轉**。
  訪客無從判斷「免費能用到什麼、付費值不值」。
- 方案結構：Free / 週 NT$33 / 月 NT$150 / 年 NT$660。
  年繳 ≈ 4.4 個月月費，折扣夠深，**應設為預設推薦以最大化 LTV**；
  週繳（≈NT$142/月）幾乎等於月費，作為「低承諾入口」尚可，但**若無免費試用，付費門檻偏高**。
- **試用是否存在待確認**（RevenueCat active_trials=0 可能只是因為幾乎沒安裝）→ 若無 trial，這是 Offer 層第一優先補洞。

### Layer 4 設計與佈局
- 落地頁層次、留白、陶土色調與宋體排版皆優秀，CTA（即刻取得/立即下載）可見度足。
- **缺社會證明**（評分/評語/下載數）——MARKETING.md 說明是刻意暫留白待數據有意義；
  但 iOS 已有 5.0（1 則）評分，可先放「App Store 5.0 ★」小徽章建立初始信任。
- 商店頁截圖（iOS 5 張，帶文字標題）情境完整（故事→語音→足跡→解鎖），品質佳。

### Layer 5 文案與訊息（最高槓桿）
落地頁 voice 極佳，但缺三塊轉換要件：
1. **異議處理缺席**——ICP 對 AI 內容的第一疑問「會不會亂編？」全頁未正面回應，
   而「以 Wikipedia 為事實依據」正是最強差異化，卻只藏在功能敘述外、未當賣點打。
2. **無 proof**——所有主張皆無可驗證佐證（可用「每則故事附 Wikipedia 出處」當 proof）。
3. **CTA 偏功能**——「即刻取得/立即下載」是動作而非結果；可測「出發就用，走到哪聽到哪」。

商店頁「行銷宣傳文字」首段已相當到位（痛點+解法+情感），描述前 3 行建議同樣以
「痛點→Wikipedia 為據的解法」開場並自然埋入 audio guide / 語音導覽 關鍵字。

## A/B 測試建議（有流量後才啟動）

1. **Hero CTA 文案**：「即刻取得」 vs 「出發就用，走到哪聽到哪」（結果框架）。
2. **異議區塊有無**：是否放「Wikipedia 為據」信任區塊對商店點擊率的影響。
3. **社會證明**：Hero 下方有/無「5.0 ★ App Store」徽章。
4. **Paywall 預設方案**：年繳預選 vs 月繳預選，看訂閱率與 LTV。
（以上在 web 新訪客達到可測樣本量前，先不啟動 A/B，避免假訊號。）

## 文案 Before / After（前 3 大修復）

**1. Hero CTA（Dict `storeButtons` / `nav.downloadApp`）**
- Before：`即刻取得` / `立即下載` / `下載 App`
- After（測）：`出發就用` / `走到哪，聽到哪` — 動作動詞 + 結果

- 內文：`每一則故事都以 Wikipedia 為事實依據，並附上出處連結——不是 AI 憑空杜撰。`

**3. 商店描述首段（App Store / Play 描述前 3 行）**
- After：`站在古蹟前，Google 只給你條列式資料。Lorescape 以 Wikipedia 史實為據，
  當場為眼前的景點寫一段有溫度的故事，再用語音說給你聽。走到哪，聽到哪。`
  （自然帶入「語音導覽 / audio guide」關鍵字）
